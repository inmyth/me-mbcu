Why do you need Functional Programming ?

Fewer bugs

Whichever the language, the most dangerous and expensive source of bugs is mutable states. These are class or object variables which are altered by some methods. What is the problem with states ? They can pass all the local tests but we cannot know if they behave correctly during runtime, especially if it's multi-threaded environment. Because of that such software is very difficult to debug. The foundation of FP is pure functions. They do not alter any state or produce side-effect. So the result is predictable. We can be sure what we have during runtime is reflected by local tests.

Scalability

These days anyone can complete a software project quickly. But usually afer the first version the pace gets slower and slower to a slog along with more modifications and features. Why does this happen ? Because developers need to dig deep into the codes to make the change, and to make sure the change does not break other things. Because in FP functions are pure they behave like Lego blocks. We can easily extend or replace them. We can also easily test it. In other word FP offers scalability. The complexity of the development would remain constant even if the software itself grows.

Suitable for Big Data

In today's world data is central to any software application. It's also presenting us with new issues: storing, transferring, transforming which cannot be solved easily due to the data size. Here are a couple scenarios where FP is useful to tackle memory and recovery problems.

Memory efficiency

One immediate issue of working with big data is that there is no machine with large enough memory to load it. What people often do is break up the data to smaller chunks which in turn causes management headache.

This video compares the old approach and the new functional stream with Java Vavr to load a very large file. In the first part, all available memory is quickly  consumed and the program crashes. But in the second, memory consumption is kept under control. This allows us to use a small machine to process very large data.

Process recovery

Imagine doing a thousand of database queries or API calls and a network failure occurs. Almost all the time this will end in the whole process being restarted, wasting hours if not days. 

Modern reactive libraries like Monix have a functionality to do task recovery. The video here shows a recovery simulation on a stream. For each element that encounters an error, it will retry at most three times or it will give up and proceed to the next element (source code). As a result we can now avoid restarting the entire process from the beginning. Using functional approach we can easily define many such strategies that would help recover a process.
